# DD-28: Project Restructure, Workspace Convention, and CLI Wrapper

**Status**: Decided
**Date**: 2026-04-15
**Depends on**: None (independent of language features)
**Blocks**: DD-29 (`lykn new` project template generation)

## Summary

Restructure the lykn project to use Deno workspace conventions,
migrate source from `src/` to `packages/lykn/`, replace
`deno.json` with `project.json` (read by the lykn CLI), add
Deno-wrapping subcommands to the lykn CLI (`run`, `test`,
`publish`), remove npm artifacts from source (generate at publish
time via `dnt`), and establish conventions that `lykn new` will
reuse.

## Motivation

1. **Single tool**: Users interact with `lykn` CLI only — no need
   to know about `deno` commands, config file names, or flags
2. **Runtime-agnostic branding**: `project.json` instead of
   `deno.json` — lykn is the language, Deno is an implementation
   detail
3. **Clean source tree**: No `package.json`, `index.d.ts`, or
   npm-specific files tracked in git — generated at publish time
4. **Workspace-ready**: Even single-package projects use
   `packages/` — consistent structure for `lykn new`
5. **Template-ready**: The conventions established here become the
   canonical template for `lykn new` (DD-29)

### Files removed from git
- `package.json` — generated in `dist/` at npm publish time
- `src/index.d.ts` — generated by `dnt` at npm publish time
- `src/index.js` — replaced by `packages/lykn/mod.js`

## project.json (workspace root)

```json
{
  "workspaces": ["./packages/lykn"],
  "tasks": {
    "test": "deno test -A test/",
    "test:unit": "deno test test/forms/ test/reader/ test/expander/",
    "test:integration": "deno test -A test/integration/",
    "build:browser": "deno run -A build.js"
  }
}
```

The workspace root defines project-wide tasks. Package-specific
config (name, version, exports, imports) lives in each package's
`deno.json`.

## packages/lykn/deno.json (package config)

```json
{
  "name": "@lykn/lykn",
  "version": "0.5.0",
  "exports": "./mod.js",
  "imports": {
    "astring": "npm:astring@^1.9.0"
  }
}
```

## mod.js (entry point)

Same content as current `index.js` but renamed to follow Deno
convention. Re-exports `read`, `expand`, `compile`, `lykn`.

Import paths in the codebase change from `./src/reader.js` to
relative paths within `packages/lykn/` (or use import map).

## CLI Subcommands

### Existing (unchanged)
```
lykn fmt <files> [--write]       Format .lykn source
lykn check <files>               Syntax check
lykn compile <file> [opts]       Compile to JavaScript
```

### New (Deno wrappers)
```
lykn run <file> [args...]        Run .lykn or .js file
lykn test [patterns...]          Run tests
lykn publish [--jsr] [--npm]     Publish package(s)
lykn lint [paths...]             Lint compiled JS
```

Each new subcommand shells out to `deno` with
`--config project.json` injected:

```rust
// lykn run main.lykn
Command::new("deno")
    .args(["run", "--config", "project.json", "-A", "main.lykn"])
    .exec()

// lykn test
Command::new("deno")
    .args(["test", "--config", "project.json", "-A", "test/"])
    .exec()

// lykn lint
Command::new("deno")
    .args(["lint", "--config", "project.json"])
    .exec()
```

### lykn publish

```rust
// lykn publish --jsr
Command::new("deno")
    .args(["publish", "--config", "project.json"])
    .exec()

// lykn publish --npm
// 1. Run dnt to generate dist/
// 2. cd dist && npm publish
// Uses dnt (Deno to Node Transform) — no npm files in source tree

// lykn publish --crates
// Existing crates.io publishing logic
```

### lykn run for .lykn files

When the target is a `.lykn` file, the CLI compiles it to a temp
`.js` file, then runs with Deno:

```rust
// lykn run main.lykn
// 1. lykn compile main.lykn -o /tmp/main.js
// 2. deno run --config project.json -A /tmp/main.js
```

## npm Publishing via dnt

At `lykn publish --npm` time:

1. Run `dnt` (Deno to Node Transform) which:
   - Reads `packages/lykn/deno.json` for metadata
   - Transpiles ESM → CJS + ESM dual output
   - Generates `package.json` from deno.json fields
   - Generates `.d.ts` type declarations
   - Outputs to `dist/npm/`
2. `cd dist/npm && npm publish --access public`

The `dnt` config can be a `build_npm.ts` script:

```typescript
import { build } from "jsr:@nicetry/dnt";

await build({
  entryPoints: ["./packages/lykn/mod.js"],
  outDir: "./dist/npm",
  shims: { deno: false },
  package: {
    name: "@lykn/lykn",
    version: "0.5.0",
    description: "S-expression syntax for JavaScript",
    license: "Apache-2.0",
    repository: { type: "git", url: "https://github.com/oxur/lykn" },
  },
});
```

## Import Path Migration

All test files and internal imports change:

| Current | Target |
|---------|--------|
| `import { read } from "../../src/reader.js"` | `import { read } from "../../packages/lykn/reader.js"` |
| `import { compile } from "../../src/compiler.js"` | `import { compile } from "../../packages/lykn/compiler.js"` |
| `import { expand } from "./src/expander.js"` | `import { expand } from "./packages/lykn/expander.js"` |

Alternative: use the import map in `project.json`:
```json
{
  "imports": {
    "lykn/": "./packages/lykn/"
  }
}
```

Then imports become: `import { read } from "lykn/reader.js"`.
This is cleaner and doesn't break if the directory structure
changes.

## Conventions for `lykn new` (DD-29)

The structure established here becomes the template:

```
<project>/
  project.json           ← workspace root
  packages/
    <name>/
      deno.json           ← package config
      mod.js              ← entry point
      mod.test.js         ← tests (optional)
  .gitignore
```

**Rules**:
- `project.json` is always at the workspace root
- Source always lives in `packages/<name>/`
- Entry point is always `mod.js`
- `packages/` is the default convention but not enforced by the
  CLI — the workspace members list in `project.json` is
  authoritative
- `lykn run`, `lykn test`, etc. resolve paths from `project.json`

## Implementation Phases

### Phase 1: File migration (no CLI changes)

1. Create `packages/lykn/` directory
2. Move `src/*.js` → `packages/lykn/` (rename `index.js` → `mod.js`,
   `lykn-browser.js` → `browser.js`)
3. Create `packages/lykn/deno.json` (package config)
4. Create `project.json` (workspace root, replaces `deno.json`)
5. Delete `deno.json`, `package.json`, `src/index.d.ts`
6. Update `.gitignore` for new paths
7. Update all import paths in test files
8. Update `build.js` for new paths
9. Update Rust CLI paths (if any reference `src/`)
10. Update `CLAUDE.md`, `README.md` for new structure
11. Verify: `deno test --config project.json -A test/` passes

### Phase 2: CLI subcommands

1. Add `run`, `test`, `lint`, `publish` subcommands to `lykn-cli`
2. Each shells out to `deno` with `--config project.json`
3. `lykn run` handles `.lykn` → compile → run pipeline
4. `lykn publish --npm` runs `dnt` + `npm publish` from dist
5. Update Makefile to use `lykn` commands where possible
6. Update README with new CLI commands

### Phase 3: npm publish via dnt

1. Create `build_npm.ts` (dnt configuration)
2. Update `lykn publish --npm` to invoke dnt
3. Remove `package.json` from git tracking
4. Remove `src/index.d.ts` from git tracking
5. Add `dist/` to `.gitignore` (already there)
6. Verify: `lykn publish --npm --dry-run` produces valid package

### Phase 4: Docs and guides

1. Update all guides that reference `src/` paths
2. Update SKILL.md document selection table
3. Update README architecture section
4. Update CLAUDE.md

## Edge Cases

| Case | Behavior |
|------|----------|
| No `project.json` found | CLI falls back to `deno.json`, then errors |
| `lykn run` on a `.js` file | Pass directly to `deno run` |
| `lykn run` on a `.lykn` file | Compile to temp, run temp |
| `lykn test` with no args | Run all tests per project.json tasks |
| `lykn publish` with no flags | Publish to JSR (default) |
| `lykn publish --npm --jsr` | Both registries |
| Multi-package workspace | Each package has own deno.json |

## Interaction with Rust crates

The Rust workspace (`Cargo.toml` + `crates/`) is unchanged. The
JS workspace (`project.json` + `packages/`) is parallel. Both
coexist at the project root. The `lykn` CLI binary is built from
`crates/lykn-cli/` and wraps both Rust compilation and Deno
commands.
