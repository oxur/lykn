# DD-50.7 Implementation Prompt for CC — Fix DD-50 Emission for Real Downstream Patterns

## Read this first

This is a **fix milestone** for compiler emission bugs that M7 / DD-50 / DD-50.5 / DD-50.5 addendum / DD-50.6 were supposed to address. Empirical disconfirmation is captured in:

- `workbench/finding-e-empirical-disconfirmation-2026-05-12.md` — the discovery memo
- `workbench/verify-finding-e-output-2026-05-12T031914Z.txt` — captured deno-check failure against `release/0.6.x` HEAD `e670088`
- `workbench/verify-finding-e-2026-05-12.sh` — reproduction script

The M7 closing report (2026-05-10) and DD-50.6 closing report (2026-05-11) both assert end-to-end correctness for the `if`-in-expression / statement classification machinery. Empirically, compiling mycelium's canonical `render.lykn` produces invalid JS that `deno check` rejects. The synthetic `compileBoth` tests covered patterns that didn't include the shape that breaks.

**This milestone closes those gaps.** It is NOT a redesign of DD-50; it is a fix to the existing machinery so that the existing design's claimed invariants actually hold for real downstream source.

You MUST treat this as a **methodology-load-bearing** milestone: the closure discipline here includes an empirical-validation gate that the prior closures lacked. See Deliverable 4 below.

## Deliverable 1 — Root-cause diagnosis (MUST in main context, MUST NOT be delegated)

Write a diagnosis memo at `workbench/dd-50.7-diagnosis-2026-05-12.md` that addresses:

### 1a. Cluster 1 root cause

The render-attrs source pattern:

```lykn
(for-of (array name value) (Object:entries attrs)
  (validate-attr-name name)
  (if (= value true)
    (swap! parts ...)
    (if (or (= value false) (js:eq value null))
      undefined
      (if (or (= (js:typeof value) "string") (= (js:typeof value) "number"))
        (swap! parts ...)
        (throw (new Error ...))))))
```

Questions to answer:

- The outer `if` is the **last** form of the `for-of` body. What context is the `for-of` body's last child supposed to receive per DD-50.5's `KernelChildProfile`?
- What context is it ACTUALLY receiving in the current emit?
- If they differ: where does the divergence happen — `KernelChildProfile` definition, kernel-form registration, surface-to-kernel expansion, or somewhere else?
- The emitted output shows nested ternary with IIFE-wrapped innermost branches. Where in the emit code does the IIFE wrapping happen, and why does its arrow body fail to wrap statements in `{...}`?
- The `return throw` artifact at line 34 — where is that emitted? Is it inside `convert_to_expression`, `emit_if_expression`, or another path? Why does the IIFE-emit not check whether its body branches are statement-typed (in which case the IIFE itself is the wrong wrapper)?

### 1b. Cluster 2 root cause

The render-element source pattern:

```lykn
(func render-element
  :args (:array el) :returns :string
  :body
  ...
  (if (is-void-element tag)
    (block
      (if (> children:length 0)
        (throw (new Error ...)))
      (return (template "<" tag attrs-str ">"))))
  (bind mapped ...)
  ...)
```

The outer `(if A B)` is in middle-of-body (statement position). It compiles correctly to `if (...) { ... }` — outer context is right. But the **inner** `(if (> children:length 0) (throw ...))` — a no-else `if` nested inside the `block` — produces a Rule 2 COMPILE_ERROR runtime-throw instead of `if (cond) throw ...;`.

Questions to answer:

- What context does `block` body assign to its children per `KernelChildProfile`?
- The inner `if` is the **first** body child of `block` (not the last). Does the profile correctly classify it as Statement?
- If the profile says Statement but the inner `if` still hits Rule 2: where does Rule 2 actually fire — is it in `emit_if_expression`, or in `convert_to_expression` called from a higher-up emit path?
- Is `block` registered with the kernel-form profile at all, or does it fall through to a default that uses Value context?

### 1c. Are clusters 1 and 2 the same root cause?

Hypothesis worth testing: both clusters look like "kernel-form body children that should be Statement context are receiving Value/Tail context, which fires Rule 2 or the IIFE-wrap path inappropriately." If true, the fix touches one or two places (`KernelChildProfile` definitions + maybe the `for-of`/`block` registration). If false, the fix is broader.

### 1d. Diagnosis MUST identify

For each cluster, the diagnosis MUST state:
- The file and approximate line where the wrong-context assignment happens.
- The file and approximate line where the resulting incorrect emit is produced.
- Whether the JS-side compiler has the same bug (verify via `compileBoth` against the relevant pattern), or if the divergence is Rust-only.
- The minimal change shape proposed for the fix, with the alternative shapes considered.

**Stop after Deliverable 1 and surface to Duncan/CDC if the diagnosis reveals:**
- The fix requires substantial restructuring of `KernelChildProfile` or context propagation (beyond adding/correcting an entry).
- The two clusters have genuinely different root causes that compound the milestone scope.
- Either compiler's existing architecture cannot represent the needed context propagation without new infrastructure.

## Deliverable 3 — Regression tests (MUST use `compileBoth`, MUST include real-downstream shapes)

File: `test/forms/dd-50.7_test.lykn` (new) — uses `compileBoth` for every test.

### 3a. Cluster 1 regression — minimal fixture

```lykn
(test "DD-50.7 cluster 1: nested if with statement branches in for-of body"
  (bind r (compile-both
    "(for-of (array k v) (Object:entries x)
       (if (= v true) (swap! parts (fn (:array p) (conj p k)))
         (if (or (= v false) (js:eq v null))
           undefined
           (if (= (js:typeof v) \"string\")
             (swap! parts (fn (:array p) (conj p k)))
             (throw (new Error \"bad value\"))))))"))
  ;; MUST NOT contain '() => if' (arrow without block braces)
  (is-equal (.includes r "() => if") false)
  ;; MUST NOT contain 'return throw' (throw wrapped in return)
  (is-equal (.includes r "return throw") false)
  ;; SHOULD contain the if/else statement structure (proper emission)
  (is-equal (or (.includes r "if (v === true)") (.includes r "if (true === v)")) true))
```

### 3b. Cluster 2 regression — minimal fixture

```lykn
(test "DD-50.7 cluster 2: no-else if in block body in func body"
  (bind r (compile-both
    "(func handle :args (:any x) :returns :void :body
       (if (some-cond x)
         (block
           (if (> x:length 0)
             (throw (new Error \"empty\")))
           (return (template \"ok\"))))
       (process-rest x))"))
  ;; MUST NOT contain COMPILE_ERROR runtime throw (Rule 2 misfiring on statement-position if)
  (is-equal (.includes r "COMPILE_ERROR") false)
  ;; SHOULD contain proper if statement inside block
  (is-equal (.includes r "if (x.length > 0) throw") true))
```
